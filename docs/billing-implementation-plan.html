Billing Implementation Plan

The companion to the monetization plan. That doc settled what we bill (tailoring credits, two tiers, overage, model-class gating, expiresFirst). This one answers how we build it: which payment rails to stand on, what data model holds the credit ledger, how a run is metered against the existing llmCalls.costUsd, how state stays in sync, and the order we ship it in.

🟢 Same workflow as before: each section ends with a field, everything auto-saves to this browser's localStorage, and Review notes compiles it to Markdown to paste back. Look for the teal EXPLORE blocks for the heavy decisions (provider, ledger, metering) — each has a recommendation and a ↳ Decide field.
🧱 Grounded in the current stack: SvelteKit · Convex · Clerk · OpenRouter · PostHog. The metering backbone already exists — llmCalls.costUsd records real per-call cost from OpenRouter, and runs.agentConfig.maxIterations already enforces loop caps. What's missing is the subscription + credit-ledger layer, which is what this plan specs.

MoR vs. Stripe, and how much to hand-roll

Two near-orthogonal questions hide in "which provider": (a) do we want a Merchant of Record to absorb global sales-tax/VAT, and (b) how much billing logic do we build vs. adopt.

A
Stripe, self-managed. Maximum control and the deepest docs. We register for tax where needed (or bolt on Stripe Tax), and we own the webhook + ledger code. Best if we expect to outgrow opinionated platforms — but tax compliance across countries is real, recurring work for a small team.
B
Merchant of Record (Polar / Paddle / Lemon Squeezy). They become the seller of record and handle VAT/sales tax worldwide for ~an extra fee. For a capital-light indie selling globally, this removes a whole compliance surface. Polar stands out for being developer-first, supporting usage-based billing, and offering a Convex integration — which maps cleanly onto a credits model.
C
Clerk Billing. Tempting because auth is already Clerk. But it's subscription-shaped; our overage/credit-pool model would still be hand-built on top, and it couples two concerns. Reasonable for the subscription half only.
Lean: a Merchant of Record, with Polar as the front-runner (MoR + usage billing + Convex-native), and Stripe as the fallback if we hit a capability wall or want maximum control. Decouple the question of "credit ledger" from "provider": we keep the ledger in Convex regardless, so the provider stays swappable. Verify each platform's current MoR terms, fees, payout countries, and Convex support before committing.

04 Build vs. buy the credit / entitlement layer

Separate from the payment rail: do we hand-write the credit ledger + feature gating, or adopt a billing-logic layer that does credits/usage/entitlements for us?

Explore

Own the ledger, or adopt a usage/entitlement layer

Our credit model has real edges: dated buckets, expiresFirst, freeze-on-cancel, overage-only-while-subscribed, one-class-up unlock. Tools like Autumn sit on top of Stripe and provide credits/usage/feature-gating; provider-native usage billing (e.g. Polar) covers part of it too.

A
Hand-roll in Convex. Full control over the exact bucket/expiry/freeze semantics, no third-party in the hot path, and Convex's transactional mutations make a correct ledger very achievable. Cost: we write & test the metering, reconciliation, and edge cases ourselves.
B
Adopt a billing-logic layer (e.g. Autumn over Stripe). Faster to a working paywall; credits, limits and gating come pre-built. Risk: our rules (freeze-on-cancel, expiresFirst across two pools, pool selection) may not map 1:1, and we add a dependency in the critical entitlement path.
C
Hybrid. Use the provider for subscriptions + one-off charges and money, but keep the credit ledger and entitlement checks in Convex. Adopt small, well-scoped helpers (SDK/component) rather than outsourcing the whole balance model.
Lean: C — hybrid. The ledger is our core billing IP and it's tightly coupled to the run lifecycle that already lives in Convex; keep it in-house and transactional. Lean on the provider/SDK only for checkout, money, and subscription state. Re-evaluate a layer like Autumn only if hand-rolling proves slow. Confirm current capabilities/pricing of any layer before adopting.
↳ Decide — build, buy, or hybrid

05 Credit ledger — the data model

The heart of the system. Credits live in dated buckets; every movement is an append-only entry. Balance is derived, never edited in place.

Explore

How to model buckets, entries & expiresFirst

We need two things at once: a fast "how many credits do I have?" read, and a tamper-evident history for reconciliation and support. The clean answer is a hybrid: mutable buckets for balance, an immutable ledger for history.

1
Buckets = each grant of credits with its own origin, amount, remaining, and expiry. A monthly subscription grant and a top-up are separate buckets. Spending decrements remaining on the bucket whose expiresAt is soonest → that is expiresFirst.
2
Ledger entries = append-only record of every grant / reserve / commit / release / expire / refund, each pointing at the bucket(s) it touched and carrying an idempotency key. Never updated, only inserted. Balance can always be re-derived by replaying entries.
3
Freeze, don't delete. On cancel, overage buckets flip to frozen (excluded from spendable balance); on resubscribe they flip back. Subscription buckets simply expire at cycle end.
Lean: hybrid buckets + append-only ledger, with Convex transactions guaranteeing atomic debits. It gives O(small) balance reads and a fully auditable trail with almost no extra cost.

Why this works on Convex
Mutations are transactional, so "pick soonest-expiring active bucket → decrement → append entry" is atomic with no row-locking ceremony. Concurrent runs can't double-spend. Cost data already exists in llmCalls, so metering reads, not writes, the source of truth.

06 Metering & debiting a run

How a run turns into credits spent — using the cost we already capture, with quiet reservation so the UI never shows a scary pre-charge.

Explore

Reserve-then-settle vs. debit-after-only

A run is multi-step (writer ↔ reviewer loops), and each llmCall only knows its true cost after it returns. We want to (a) never let a run spend credits the user doesn't have, and (b) never show anxiety-inducing pre-charges.

A
Reserve → settle (quiet). At run start, place a soft reservation for an estimated cost (a reserve ledger entry) so balance can't be double-spent by a parallel run. After each loop, read the real llmCalls.costUsd, commit the actual credits, and release the unused remainder. Before each new loop, check affordability. UI shows "~N runs left" and the actual cost calmly after.
B
Debit-after-only. Simplest: let the run go, sum costUsd at the end, debit once. Risk: a user at near-zero balance can overspend on an expensive run, and concurrent runs race. Fine for an MVP with generous balances, weak once margins matter.
Lean: A — reserve→settle, but the reservation is internal (quiet). Reserve a conservative per-model estimate, settle to actual per loop, and gate the next loop on real remaining balance. This is exactly the "quiet reservation + affordability check per loop" the monetization plan settled on. Ship B first only if it accelerates the MVP, then upgrade.
↳ Decide — metering model + how conservative the reservation is

The debit formula & lifecycle

credits = ceil( providerCostUsd × markup ÷ creditUsdValue )

providerCostUsd = sum of llmCalls.costUsd for the run/loop (already populated from OpenRouter usage accounting). markup and creditUsdValue are config, calibrated once the cost-logging data matures.

on runStart(userId, model, plan):
  assert subscription.active && model in entitlements(plan, pools)
  est = estimateCost(model, plan)          // rolling avg from llmCalls until data lands
  reserve(userId, est)                      // quiet; ledger entry kind="reserve"

on loopEnd(runId, loopNumber):
  actual = sum(llmCalls.costUsd where runId, loopNumber)
  commit(userId, creditsFor(actual))        // kind="commit", expiresFirst buckets
  if !canAfford(nextLoopEstimate): stop_with_grace()  // affordability gate

on runEnd | runFailed:
  release(remaining_reservation)            // kind="release"; failed loops aren't charged
Edge to nail
Charge only for settled work. If a loop errors before producing output, release its reservation — don't bill failed inference to the user. The existing llmCalls.status + attemptNumber fields make "what actually succeeded" queryable.

08 Webhooks & state sync

The provider tells us what happened; we react idempotently. This is where most billing bugs live, so it gets explicit rules.

Flow

  • Provider events hit a Convex httpAction endpoint.
  • Verify the signature before doing anything; reject unsigned/old events.
  • Dedupe on the provider event id (idempotency) — webhooks are at-least-once, so the same event can arrive twice.
  • Translate event → ledger/subscription change inside a transaction.

Events → actions

Event (conceptual)Action
checkout / subscription createdCreate billingCustomers + subscriptions; grant first month's bucket.
invoice paid (renewal)Expire last cycle's subscription bucket; grant the new cycle's bucket. Idempotent on invoice id.
subscription updated (plan change)Update mirror; adjust entitlements; handle proration policy.
payment failed / past_dueFlip status; entitlement gate tightens; dunning handled by provider.
subscription canceledStatus canceled; freeze overage buckets; subscription bucket expires at period end.
one-off purchase paidMark purchases.paid; grant overage/bundle bucket.
refund / chargebackMark refunded; claw back unspent credits (see Integrity).
Golden rule
Credit grants are driven by paid webhooks, never by the client returning from checkout. The client redirect is just UX; money-truth comes from the signed webhook.

09 One-off purchases — overage & bundles

Top-ups and (later) bundles. The infra must handle one-off charges, not just recurring — a requirement flagged in the monetization plan.

  • Overage top-ups: only offered to active subscribers within the cycle. The buy button is gated by the entitlements resolver. Purchase → paid webhook → overage bucket (rolls forward, ~6-mo expiry).
  • Per-tier purchase caps: enforce "can't buy more than this tier allows" at purchase-create; higher tiers unlock larger packs.
  • Bundles (later): a clearly-labelled secondary product with a higher per-credit price and 30–60-day expiry. Same purchases + bucket machinery, different catalog entry. Lower priority — ship after the core loop is proven.
  • Pool selection (still to think through): letting a user choose which pool funds a run is a UI + debit-time concern; the bucket model already supports it (pick bucket by source instead of pure expiresFirst).
Reuse, don't fork
Subscriptions, overage, and bundles are all "grant a bucket on a paid event." One purchase pipeline, one bucket model, three catalog entries — keeps the infra honest.

10 Pricing configuration — keeping it flexible

Every price, grant, markup and class-map is config, never a hardcoded literal. The monetization plan insists pricing stays movable; this is how we honour that in code.

  • One catalog defines plans, credit grants, packs, creditUsdValue, markup, and the model-class map. Provider price IDs map to catalog entries, not the reverse.
  • Effective-dated & versioned: changing a price creates a new version; existing subscribers keep their grandfathered terms until renewal. The ledger records which catalog version a grant used.
  • Code config vs. Convex table: a typed config in the repo is simplest and reviewable; a Convex table allows live edits without deploy. Likely start in-code, move hot knobs (grants, markup) to a table once we're tuning against real cost data.
Why it matters
When the cost-logging data lands and we finally set "how many credits / how much," it should be a config change + a catalog version — not a refactor.

11 Cost calibration loop

The bridge to the one still-open monetization question ("how much / how many credits"). Implementation makes the numbers derivable, not guessed.

  1. The LLM-cost work (already landing — llmCalls.costUsd populated per call) gives us real per-run cost across model classes.
  2. Aggregate to avg / P95 / worst-case cost per completed run, per class — push these to PostHog and/or a Convex aggregate.
  3. Set creditUsdValue, markup (keeping the inference budget deliberately <50%), and per-tier grants from those numbers.
  4. Set reservation estimates and per-loop affordability thresholds from the same distribution (reserve near P95, settle to actual).
Dependency
This is the BLOCKER for final pricing — but not for building the machinery. We build the ledger, metering, and gating against config now, and the cost data only fills in the config later.

12 Integrity, reconciliation & abuse

Money code must reconcile and resist gaming. The append-only ledger makes most of this cheap.

  • No double-spend: debits run in Convex transactions; concurrent runs serialize naturally. Each debit carries an idempotency key so a retried mutation can't charge twice.
  • Invariant checks: sum of bucket remaining == balance derived from replaying ledgerEntries. A scheduled job asserts this and alerts on drift.
  • Provider reconciliation: periodically compare Convex subscriptions/purchases against the provider to catch missed/late webhooks; re-sync the diff.
  • Clawback: on refund/chargeback, reverse the matching grant; if those credits were already spent, allow a negative balance and block runs until cleared (policy decision below).
  • Abuse: the strongest control is the existing one — account + email verification + must-subscribe, so there's no anonymous free inference to farm. Credits are the rate limit.

15 Phased rollout — the build order

Ship value early, defer complexity. Each phase is independently useful and testable.

Explore

How aggressively to phase

The tension: a simple paywall (subscribe → flat monthly credits, debit-after) could ship fast and start revenue; the full reserve/settle ledger is more work. We can sequence so the simple version is a true subset of the final one — no throwaway.

Lean: ship subscriptions + gating + a working (even if approximate) credit debit first to start revenue and learn, then layer reserve/settle, overage, and bundles. Crucially, build the ledger tables from Phase 1 even if Phase 1 only does debit-after — so later phases extend, never rewrite.
